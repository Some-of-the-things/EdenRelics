About This Report

This report summarises the results of an accessibility audit of the Eden Relics website. The audit checks whether the site can be used comfortably by everyone, including people who rely on screen readers, keyboard navigation, or other assistive technologies.

We tested seven key pages — the home page, login, registration, contact form, blog, shopping cart, and privacy policy — using a combination of automated scanning tools and a detailed manual review of the underlying code.

The site was assessed against WCAG 2.1 Level AA, the internationally recognised standard for web accessibility adopted by governments and organisations worldwide. Where technical criteria are referenced, they are listed in the appendix at the end of this report for those who wish to explore further.

Overall Assessment

Verdict: Strong foundation with a small number of improvements needed The Eden Relics website already includes many important accessibility features. The automated scan returned zero errors across all seven pages tested, which places it well ahead of the majority of e-commerce sites. A handful of areas identified during the manual review need attention to reach full compliance.

What the Site Does Well

Before looking at what can be improved, it is worth highlighting the many things Eden Relics already gets right. These features show a genuine commitment to making the site usable for everyone:

  • Skip navigation — a hidden link lets keyboard users jump straight past the menu to the main content, saving time on every page.
  • Clear page structure — the site uses proper HTML elements (headings, navigation landmarks, main content area) so screen readers can present a logical outline of each page.
  • Image descriptions — all product images and the site logo have text descriptions that are read aloud by screen readers.
  • Visible focus indicator — when navigating with a keyboard, a clear outline shows which element is currently selected.
  • Proper form labels — every form field (name, email, password, etc.) has an associated label so screen reader users know what information to enter.
  • Browser auto-fill support — form fields include hints that allow browsers to auto-fill personal details, which is especially helpful for users with motor difficulties.
  • Meaningful page titles — each page has a unique title (e.g. "Blog | Eden Relics") so users with multiple tabs open can distinguish between them.
  • Readable fonts — text uses the system font with font-display: swap, which prevents invisible text while custom fonts load.
  • Responsive images — images include width and height attributes that prevent the page from jumping around as content loads.

Areas for Improvement

The following areas were identified as needing attention. They are listed in order of priority, starting with the most impactful changes.

1. Pop-up windows need better keyboard support

High priority

When a pop-up window (for example, the prompt to sign in before adding a favourite) appears on screen, keyboard-only users can accidentally move focus to elements behind it. There is also no way to close the pop-up by pressing the Escape key, which is the expected behaviour.

What this means for users: Someone navigating with a keyboard or screen reader may become confused or stuck when a pop-up is displayed, as their focus can drift to hidden content behind it.

Recommended fix: Keep keyboard focus trapped within the pop-up while it is open, add Escape key support to close it, and return focus to the button that triggered it when it closes.

2. Dynamic updates are not announced to screen readers

High priority

When something changes on the page without a full reload — for example, adding an item to the shopping cart, filtering products by category, or a form being submitted — screen reader users are not notified of the change. They may not realise that their action had any effect.

What this means for users: A screen reader user who adds a product to their cart might not hear any confirmation, leaving them unsure whether the action succeeded.

Recommended fix: Add invisible announcement regions that screen readers monitor for updates. When the cart count changes or a filter is applied, a short message (e.g. "Item added to cart" or "Showing 12 results") should be announced.

3. Text contrast on product image overlays needs checking

Medium priority

Product cards display a decade label (e.g. "1970s") overlaid on the product photograph. Because the label sits on top of a varying image, the contrast between the text and the background can differ from product to product. The automated scanner flagged 20 instances where it could not confirm sufficient contrast.

What this means for users: People with low vision or colour deficiencies may struggle to read the decade label on some product cards, particularly where the product image is light-coloured.

Recommended fix: Add a solid or high-opacity background behind the decade label to guarantee readable contrast regardless of the underlying image.

4. Form error messages could be more descriptive

Medium priority

When a user submits a form with missing or incorrect information, an error message appears on screen. While the message is announced by screen readers, it is not directly linked to the specific field that needs correcting. On longer forms (such as the checkout), this can make it harder to find and fix the problem.

What this means for users: A screen reader user filling in the checkout form might hear "Please check your details" without knowing whether the issue is with their name, email, or address.

Recommended fix: Link each error message to its corresponding form field so that screen readers can announce the error when the user navigates to that field.

5. Some buttons could be larger for touchscreen users

Medium priority

The favourite (heart) button on product cards measures 36×36 pixels. While this meets the baseline requirement, the recommended minimum for comfortable use on touchscreens is 44×44 pixels. Some navigation links may also be small on mobile devices.

What this means for users: People with motor difficulties or those using a mobile phone may find it harder to accurately tap smaller buttons.

Recommended fix: Increase the size of small interactive elements to at least 44×44 pixels, or add extra padding around them.

6. Decorative icons should be hidden from screen readers

Low priority

Some decorative icons (such as the heart icon on the favourite button and the social login provider logos) are not explicitly marked as decorative. Screen readers may attempt to describe these images, which adds unnecessary noise.

Recommended fix: Mark decorative icons as hidden from assistive technology, and ensure that icon-only buttons (such as "Sign in with Google") have a text label that screen readers can announce.

Appendix A — WCAG 2.1 Criteria Reference

The findings in this report are measured against the Web Content Accessibility Guidelines (WCAG) 2.1, Level AA. The table below maps each finding to the specific success criterion it relates to.

Finding WCAG Criterion Description
Pop-up keyboard support 2.4.3 Focus Order Focusable components receive focus in an order that preserves meaning and operability.
Screen reader announcements 4.1.3 Status Messages Status messages can be programmatically determined so that assistive technologies can present them without receiving focus.
Text contrast on overlays 1.4.3 Contrast (Minimum) Text has a contrast ratio of at least 4.5:1 (3:1 for large text).
Form error descriptions 3.3.1 Error Identification If an input error is detected, the item in error is identified and described to the user in text.
Form error descriptions 3.3.3 Error Suggestion If an input error is detected and suggestions are known, they are provided to the user.
Touchscreen button sizes 2.5.5 Target Size (AAA) Interactive targets are at least 44 by 44 CSS pixels. (Level AAA recommendation; 2.5.8 at Level AA requires 24px.)
Decorative icon labelling 4.1.2 Name, Role, Value All user interface components have accessible names and roles that can be programmatically determined.
Decorative icon labelling 1.1.1 Non-text Content All non-text content has a text alternative, or is marked as decorative.
